# RSplitCap 设计方案
## 1. 项目概述
### 1.1 项目背景
SplitCap 是一款经典的 PCAP 文件拆分工具，支持按会话、IP、MAC、时间等维度拆分抓包文件，但存在原生不支持 PCAP-NG 格式、拆分后生成大量零散文件、无法快速随机提取指定流等局限性。

本项目使用 Rust 开发 RSplitCap，在 100% 兼容 SplitCap 原有功能与命令行接口的基础上，原生支持 PCAP/PCAP-NG 多格式，新增单文件归档与快速索引提取能力。**归档格式采用「数据包顺序存储 + 流偏移索引」的架构**，从根本上解决大文件场景下的内存爆炸与随机IO问题，充分利用 Rust 的内存安全、零拷贝、并发等特性实现性能与稳定性的全面提升。

### 1.2 设计目标
1. **完全兼容**：命令行参数、拆分逻辑、输出格式与原 SplitCap 保持一致，支持用户无缝迁移
2. **格式扩展**：原生支持 PCAP、PCAP-NG 格式解析，无需依赖外部格式转换工具
3. **归档能力**：新增自定义 `.rsplit` 单文件归档格式，采用流式写入架构，支持百GB级大文件低内存处理
4. **高效提取**：支持通过流 ID、五元组、IP/端口快速定位并提取指定流，基于内存映射实现高效随机读取
5. **性能优化**：充分发挥 Rust 特性，在大文件处理速度、内存占用、并发能力上优于原版 SplitCap
6. **跨平台**：纯 Rust 实现，原生支持 Windows / Linux / macOS，无需 Mono 等运行时依赖

## 2. 功能需求
### 2.1 兼容 SplitCap 核心功能
#### 拆分分组维度（`-s` 参数）
完全支持原 SplitCap 的所有分组策略：
- `session`（默认）：按双向会话分组，同一 TCP/UDP 会话的双向流量归入同一文件
- `flow`：按单向五元组（传输协议+源目IP+源目端口）分组
- `host`：按单个 IP 地址分组，每个 IP 对应一个文件，数据包会同时归入源、目的两个文件
- `hostpair`：按通信 IP 对分组，同一对 IP 的双向流量归入同一文件
- `mac`：按 MAC 地址分组，规则同 host
- `bssid`：按 WiFi 网络 BSSID 分组
- `seconds <s>`：按时间分片，每 s 秒生成一个新文件
- `packets <c>`：按包数分片，每 c 个数据包生成一个新文件
- `nosplit`：不拆分，仅做过滤后输出单个文件

#### 过滤能力
- 支持多 IP 过滤（`-ip` 参数可重复指定）
- 支持多端口过滤（`-port` 参数可重复指定）
- 过滤性能优于 tshark，支持大文件流式过滤

#### 输出格式（`-y` 参数）
- `pcap`：输出完整 PCAP 帧（默认）
- `L7`：仅提取并保存应用层 payload 数据

#### 基础能力
- 支持指定输入文件（`-r`），支持 `-` 从标准输入读取
- 支持指定输出目录（`-o`）
- 支持处理前清空输出目录旧数据（`-d`）
- 支持控制内存中最大并发会话数（`-p`），超出阈值自动淘汰不活跃流
- 支持设置单文件缓冲区大小（`-b`），批量写入减少磁盘 IO
- 支持递归处理目录下所有抓包文件（`-recursive`）

### 2.2 新增核心功能
1. **多格式原生支持**：内置 PCAP、PCAP-NG 解析器，无需 editcap 转换即可直接处理 pcapng 文件
2. **流式单文件归档**：支持将拆分后的所有流写入单个 `.rsplit` 归档文件，采用顺序写入+后置索引架构，百GB文件内存占用稳定在百MB级
3. **多维度快速流提取**：支持从归档文件中通过流 ID、五元组、IP、端口、协议快速筛选并提取对应流的 PCAP 数据
4. **流元数据查询**：支持列出归档文件中所有流的统计信息（五元组、包数、时间范围、字节数）
5. **内存映射加速**：大文件自动启用 mmap 机制，降低内存拷贝与系统调用开销

### 2.3 非功能需求
- 性能：相同场景下处理速度不低于原版 SplitCap，大文件场景提升 30% 以上
- 内存：10GB 级文件处理内存占用控制在 200MB 以内，百GB级文件内存占用不随文件大小线性增长
- 稳定性：畸形文件、超大文件场景下无崩溃、无内存泄漏
- 兼容性：命令行参数与 SplitCap 完全兼容，原有脚本可直接替换二进制使用

## 3. 整体架构设计
采用分层模块化架构，各模块职责单一、边界清晰，便于扩展与测试。

```
┌─────────────────────────────────────────────────────────┐
│                     CLI 接口层                          │
│  参数解析、模式分发（拆分模式 / 归档模式 / 提取模式）   │
└─────────────────────┬───────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────┐
│                     核心调度层                          │
│  流水线调度、线程池管理、流控与内存管控                 │
└─────────┬───────────────────────┬───────────────────────┘
          │                       │
┌─────────▼─────────┐   ┌─────────▼─────────┐   ┌─────────▼─────────┐
│  数据包解析层     │   │  流分类过滤层     │   │  输出层           │
│  - PCAP 解析      │   │  - 分组策略       │   │  - 拆分PCAP输出   │
│  - PCAP-NG 解析   │   │  - IP/端口过滤    │   │  - L7数据输出     │
│  - 协议元数据提取 │   │  - 流表管理       │   │  - .rsplit归档    │
│  - Stdin 支持     │   │  - LRU淘汰        │   │  - 索引构建       │
└───────────────────┘   └───────────────────┘   └───────────────────┘
```

### 核心数据流
1. 输入文件/标准输入 → 解析层统一转换为标准 Packet 结构
2. 过滤层根据规则丢弃不匹配数据包
3. 分类层根据分组策略计算分组 Key，更新流表元数据
4. **拆分模式**：输出层按分组批量写入独立 PCAP 文件
5. **归档模式**：输出层顺序写入数据包到 PacketRegion，同步记录每个流的数据包偏移；全文件处理完成后集中写入索引区与文件尾
6. **提取模式**：读取归档文件索引，通过偏移定位数据包，组装后输出

## 4. 核心模块详细设计
### 4.1 数据包解析模块 `packet_parser`
#### 职责
统一封装 PCAP/PCAP-NG 解析逻辑，向上输出标准化的数据包结构，屏蔽底层格式差异。

#### 核心设计
- 定义统一 `CaptureReader` trait，所有解析器实现该接口
- 基于零拷贝思想解析，数据包原始数据使用 `Bytes` 持有，避免内存拷贝
- 自动识别文件格式（通过魔数判断 PCAP/PCAP-NG），无需用户指定

```rust
/// 统一捕获文件读取器 trait
pub trait CaptureReader {
    /// 获取链路层类型
    fn link_type(&self) -> u32;
    /// 读取下一个数据包，返回 None 表示文件结束
    fn next_packet(&mut self) -> Result<Option<Packet>, ParseError>;
}
```

#### 统一数据包结构
```rust
use bytes::Bytes;
use std::net::IpAddr;
use std::time::SystemTime;

/// 五元组标识
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct FiveTuple {
    pub protocol: u8,
    pub src_ip: IpAddr,
    pub dst_ip: IpAddr,
    pub src_port: u16,
    pub dst_port: u16,
}

/// 标准化数据包
#[derive(Debug, Clone)]
pub struct Packet {
    pub timestamp: SystemTime,
    pub original_len: u32,
    pub captured_len: u32,
    pub data: Bytes,          // 原始帧数据，零拷贝
    pub src_mac: Option<[u8; 6]>,
    pub dst_mac: Option<[u8; 6]>,
    pub bssid: Option<[u8; 6]>,
    pub five_tuple: Option<FiveTuple>,
}
```

#### 依赖选型
- 协议头解析：`pnet_packet`，支持以太网、IPv4/IPv6、TCP/UDP 等多层协议解析
- PCAP 解析：基于 `nom` 自定义实现零拷贝解析，或使用 `pcap-file` crate
- PCAP-NG 解析：基于 `pcapng` crate 扩展，支持增强包块、接口描述块等核心块类型

### 4.2 流分类与管理模块 `flow_manager`
#### 职责
实现所有分组策略，维护流表元数据与偏移列表，管理流的生命周期与内存淘汰。

#### 分组策略设计
定义 `GroupStrategy` trait，每种分组维度实现该 trait，便于扩展与测试。

```rust
pub trait GroupStrategy {
    type Key: Hash + Eq + Clone;
    /// 计算数据包对应的分组 Key
    fn group_key(&self, packet: &Packet) -> Option<Self::Key>;
}
```

典型实现：
- `SessionStrategy`：对五元组做排序后生成 Key，双向流量归入同一会话
- `FlowStrategy`：直接使用原始五元组作为 Key
- `HostStrategy`：分别返回源 IP、目的 IP 两个 Key，数据包写入两个分组
- `TimeStrategy`：按时间戳取整作为 Key，实现时间分片
- `PacketCountStrategy`：按包数计数器滚动生成 Key

#### 流表与内存管控
- **流状态存储**：每个流仅维护元数据与偏移列表，不缓存数据包本体
  ```rust
  pub struct FlowState {
      pub flow_id: u64,
      pub five_tuple: FiveTuple,
      pub packet_count: u64,
      pub total_bytes: u64,
      pub start_ts: SystemTime,
      pub end_ts: SystemTime,
      pub packet_offsets: Vec<u64>, // 该流所有包在归档文件中的偏移
  }
  ```
- 基础存储：使用 `hashbrown` + `fxhash` 高性能哈希表存储流状态
- 并发场景：按 Key 哈希分片到多个子流表，降低锁竞争；或使用 `dashmap` 并发哈希表
- LRU 淘汰：维护流的访问时序，当流数量超过 `-p` 参数阈值时，淘汰最久未访问的流；归档模式下可将偏移列表暂存到临时文件，释放内存
- 偏移压缩：内存中可采用增量方式存储偏移，降低内存占用

### 4.3 过滤模块 `filter`
#### 职责
在数据包解析完成后、进入流表前执行过滤，提前丢弃不匹配数据，降低后续处理开销。

#### 能力
- 支持多 IP 白名单过滤：匹配源或目的 IP 任一即可通过
- 支持多端口白名单过滤：匹配源或目的端口任一即可通过
- IP 与端口过滤为 AND 关系，同时满足才通过
- 支持无过滤模式，全部数据包放行

### 4.4 输出模块 `output`
#### 4.4.1 拆分 PCAP 输出
- 兼容原版 SplitCap 的文件命名规则与目录结构
- 每个分组对应一个输出文件，内置缓冲区批量写入
- 支持流淘汰时自动关闭文件，新数据包到达时重新打开追加写入
- 输出文件为标准 PCAP 格式，可直接用 Wireshark 打开

#### 4.4.2 L7 应用层输出
- 剥离链路层、网络层、传输层头部，仅保存传输层 payload
- 按流连续写入，保留数据时序

#### 4.4.3 自定义归档输出（.rsplit）
采用「两阶段写入」架构：

**第一阶段：流式写入阶段**
1. 写入文件头（Header）
2. 逐包读取解析后，直接顺序写入 PacketRegion
3. 记录每个数据包写入后的文件偏移，追加到对应流的 `packet_offsets` 列表
4. 更新流的统计信息（包数、字节数、起止时间）
5. 全程纯顺序写入，无文件回溯操作，内存仅保留流元数据与偏移列表

**第二阶段：收尾索引阶段**
1. 所有数据包写入完成后，进入索引构建阶段
2. 依次写入 FlowPacketRegion（所有流的偏移列表）、FlowTable（流元数据表）、二级索引区
3. 写入 Footer，记录所有区域的偏移与大小
4. 单遍收尾，无随机IO

## 5. 自定义归档格式（.rsplit）设计
### 5.1 设计原则
1. **流式写入优先**：数据包纯顺序写入，全程无随机回写，支持管道输入与超大文件低内存处理
2. **单文件聚合**：所有流数据与索引整合到单个文件，便于传输与管理
3. **索引后置构建**：所有索引在数据写入完成后集中生成，写入性能最优
4. **提取高效**：基于内存映射实现零拷贝索引读取，通过偏移快速定位数据包
5. **元数据丰富**：内置流元数据与二级倒排索引，支持多维度快速筛选
6. **兼容标准**：数据包采用标准 PCAP 帧格式，提取后可直接组装为合法 PCAP 文件
7. **可扩展**：通过 Footer 统一管理索引入口，新增索引类型无需修改文件头结构

### 5.2 文件整体布局
文件按写入顺序分为 7 个区域，所有索引区后置，Footer 固定在文件末尾。

```
+---------------------+  ← 文件起始 (offset 0)
|   File Header       |  固定长度，全局基础元信息
+---------------------+
|                     |
|   Packet Region     |  变长，顺序存储所有数据包（标准PCAP帧格式）
|                     |
+---------------------+
|                     |
| Flow Packet Region  |  变长，存储每个流对应的数据包偏移列表
|                     |
+---------------------+
|                     |
|     Flow Table      |  变长，流元数据表，每个流一条目
|                     |
+---------------------+
|                     |
|  Secondary Indexes  |  变长，二级倒排索引（IP/端口/协议）
|                     |
+---------------------+
|   File Footer       |  固定长度，所有区域的偏移与大小入口
+---------------------+  ← 文件结束
```

### 5.3 文件头（File Header）
固定 64 字节，位于文件起始位置，存储全局基础信息。

| 字段 | 类型 | 长度(字节) | 说明 |
|------|------|------------|------|
| magic | [u8; 8] | 8 | 魔数：`b"RSPLIT02"` |
| version | u32 | 4 | 格式版本号（当前为 2） |
| link_type | u32 | 4 | 全局链路层类型，与 PCAP 定义一致 |
| ts_resolution | u8 | 1 | 时间戳精度：0=微秒，1=纳秒 |
| flags | u8 | 1 | 标志位：预留压缩、加密等标识 |
| reserved | [u8; 46] | 46 | 保留字段，对齐到 64 字节 |

### 5.4 数据包区（Packet Region）
- 所有数据包严格按接收时间顺序连续存储
- 单条数据包采用**标准 PCAP 帧记录格式**，与原始 PCAP 文件格式完全兼容：
  ```
  ts_sec (u32) → ts_usec (u32) → incl_len (u32) → orig_len (u32) → 帧数据
  ```
- 优势：提取时直接拷贝对应偏移的数据，加上 PCAP 全局头即可生成合法 PCAP 文件，无需重组解析
- 写入特性：纯顺序追加，无回溯，支持流式输入与增量追加

### 5.5 流偏移区（Flow Packet Region）
存储每个流对应的所有数据包在 Packet Region 中的文件偏移列表。

#### 存储格式
- 每个流的偏移列表连续存储
- 采用 **Delta + LEB128 变长编码** 压缩：
  1. 第一个偏移存原始值（u64）
  2. 后续每个偏移存储与前一个偏移的差值
  3. 差值使用无符号 LEB128 变长编码，平均每个偏移仅需 2~3 字节
- 压缩收益：5000 万包的索引体积可从 400MB（u64）压缩至 100MB 以内，占比不足原始流量的 0.1%

### 5.6 流表区（Flow Table）
存储所有流的元数据条目，采用定长条目存储，支持快速遍历与随机访问。

#### 单条流条目结构（FlowEntry）
```rust
#[derive(Debug, Clone, Copy)]
#[repr(C)]
pub struct FlowEntry {
    pub flow_id: u64,        // 流唯一ID
    pub protocol: u8,        // 传输协议号: 6=TCP, 17=UDP
    pub src_ip: [u8; 16],    // IPv4映射为IPv6兼容格式存储，统一16字节
    pub dst_ip: [u8; 16],    // 目的IP
    pub src_port: u16,       // 源端口
    pub dst_port: u16,       // 目的端口
    pub offset_list_offset: u64, // 该流偏移列表在 FlowPacketRegion 中的起始偏移
    pub offset_list_size: u32,   // 偏移列表编码后的字节长度
    pub packet_count: u32,       // 数据包总数
    pub start_ts_us: u64,        // 首包时间戳(微秒)
    pub end_ts_us: u64,          // 末包时间戳(微秒)
    pub total_bytes: u64,        // 总字节数(原始长度)
}
```
- 条目固定长度：96 字节，支持按流 ID 直接计算偏移随机读取
- IP 统一存储为 16 字节，兼容 IPv4/IPv6

### 5.7 二级索引区（Secondary Indexes）
可选的倒排索引，用于快速按 IP、端口、协议筛选流，提升复杂查询性能。

#### 包含三类索引
1. **IP 索引**：`IP地址 → 流ID列表`，支持快速查找指定 IP 参与的所有流
2. **端口索引**：`端口号 → 流ID列表`，支持快速查找指定端口的所有流
3. **协议索引**：`协议号 → 流ID列表`，支持快速筛选 TCP/UDP/ICMP 等流量

#### 存储格式
- 索引 Key 排序存储，支持二分查找
- Value 为流 ID 列表，采用有序数组存储，可快速求交集（如同时筛选 IP+端口）

### 5.8 文件尾（File Footer）
固定 128 字节，位于文件末尾，是读取归档文件的统一入口。打开文件时仅需读取末尾 128 字节，即可定位所有索引区域。

| 字段 | 类型 | 长度(字节) | 说明 |
|------|------|------------|------|
| magic | [u8; 8] | 8 | 尾部魔数：`b"RSPLFOOT"` |
| version | u32 | 4 | 格式版本号 |
| packet_region_offset | u64 | 8 | Packet Region 起始偏移 |
| packet_region_size | u64 | 8 | Packet Region 总字节数 |
| flow_packet_offset | u64 | 8 | Flow Packet Region 起始偏移 |
| flow_packet_size | u64 | 8 | Flow Packet Region 总字节数 |
| flow_table_offset | u64 | 8 | Flow Table 起始偏移 |
| flow_table_size | u64 | 8 | Flow Table 总字节数 |
| flow_count | u64 | 8 | 总流数量 |
| secondary_index_offset | u64 | 8 | 二级索引区起始偏移 |
| secondary_index_size | u64 | 8 | 二级索引区总字节数 |
| reserved | [u8; 64] | 64 | 保留字段，对齐到 128 字节 |

### 5.9 写入完整流程
#### 阶段1：流式数据写入
1. 写入 64 字节 File Header（预留字段填充 0）
2. 记录 Packet Region 起始偏移
3. 循环读取输入数据包：
   - 解析过滤后，将 PCAP 帧格式的数据包写入文件
   - 记录该包的起始文件偏移
   - 根据分组策略找到对应流，将偏移追加到流的偏移列表
   - 更新流的统计信息（包数、字节、起止时间）
4. 所有数据包处理完成，记录 Packet Region 结束位置

#### 阶段2：索引构建与收尾
1. 记录 Flow Packet Region 起始偏移
2. 遍历所有流：
   - 对该流的偏移列表执行 Delta + LEB128 编码
   - 将编码后的数据写入文件
   - 在 FlowEntry 中记录该流偏移列表的偏移与大小
3. 记录 Flow Table 起始偏移，依次写入所有 FlowEntry
4. 构建二级索引并写入文件
5. 填充 File Footer 所有字段，写入文件末尾
6. 刷盘完成，归档文件生成结束

### 5.10 提取完整流程
#### 单流提取（按 Flow ID）
1. 打开文件，读取末尾 128 字节 Footer，获取各区域偏移
2. mmap 映射 Flow Table 区域，根据 Flow ID 定位对应 FlowEntry
3. 读取该流的偏移列表，解码得到所有数据包的文件偏移
4. mmap 映射 Packet Region，根据偏移读取每个数据包
5. 为输出数据添加 PCAP 全局头，写入目标文件或标准输出

#### 条件筛选提取（按 IP/端口/协议）
1. 读取 Footer，加载二级索引
2. 根据筛选条件在对应索引中查找匹配的流 ID 列表
3. 多条件时对流 ID 列表求交集
4. 对每个匹配的流执行单流提取流程，批量输出到指定目录

## 6. Rust 特性与性能优化
### 6.1 零拷贝与内存效率
1. **全链路零拷贝**：从文件读取、协议解析到写入，使用 `Bytes` 智能指针传递数据，避免冗余拷贝；归档提取时基于 mmap 直接返回文件切片，零拷贝输出
2. **内存映射**：大文件自动启用 `memmap2` 内存映射，减少内核态到用户态的拷贝开销；提取模式下索引与数据均通过 mmap 访问，无需加载全量索引到堆内存
3. **紧凑数据结构**：FlowEntry 采用 `repr(C)` 定长布局，避免内存对齐浪费；偏移列表采用 Delta + LEB128 压缩，索引体积降低 70% 以上
4. **池化复用**：数据包缓冲区、字节数组使用对象池复用，减少频繁分配开销

### 6.2 流式写入与架构优化
1. **纯顺序写入**：归档模式下数据包区无任何随机回写，磁盘 IO 性能拉满，充分发挥顺序写入的带宽优势
2. **内存与文件大小解耦**：内存占用仅与并发流数量相关，不随文件总大小线性增长，百GB级文件可在 200MB 内存内完成处理
3. **三级流水线**：读取解析 → 分类过滤 → 写入输出，通过 `crossbeam-channel` 传递数据，各阶段并行执行
4. **批量处理**：按批次处理数据包，批量写入磁盘、批量更新偏移，降低系统调用与锁获取频率

### 6.3 并发与多核利用
1. **无锁流表分片**：按分组 Key 哈希分片到多个独立流表，避免全局锁，大幅提升并发吞吐量
2. **多线程解析**：大文件下可启用多线程分块解析，充分利用多核 CPU
3. **并行索引构建**：收尾阶段索引构建可并行执行，缩短大文件处理的收尾耗时

### 6.4 安全与稳定性
1. **内存安全**：完全利用 Rust 所有权与借用检查，避免野指针、缓冲区溢出等内存问题
2. **健壮解析**：所有解析逻辑做严格边界检查，畸形文件仅返回错误，不崩溃、不 panic
3. **错误处理**：基于 `thiserror` + `anyhow` 构建分层错误体系，所有异常场景有明确处理
4. **原子性写入**：输出文件采用临时文件+重命名模式，避免程序中断产生损坏文件

## 7. CLI 接口设计
### 7.1 兼容 SplitCap 原生参数
与原版 SplitCap 参数完全一致，原有脚本可直接替换使用。

```
USAGE:
    rsplitcap [OPTIONS]

OPTIONS:
    -r <input_file>          输入pcap/pcapng文件，"-"表示从标准输入读取
    -o <output_directory>    指定输出目录
    -d                       处理前删除输出目录内的历史数据
    -p <nr_parallel_sessions> 内存中保留的最大并发会话数 (默认: 10000)
    -b <file_buffer_bytes>   单个输出文件的缓冲区字节数 (默认: 10000)
    -s <GROUP>               分组拆分策略:
        bssid                按WLAN BSSID分组
        flow                 按单向五元组流分组
        host                 按单个IP主机分组
        hostpair             按通信IP对分组
        mac                  按MAC地址分组
        nosplit              不拆分，仅过滤后输出单文件
        session              按双向会话分组 (默认)
        seconds <s>          按时间分片，每s秒生成新文件
        packets <c>          按包数分片，每c个包生成新文件
    -ip <IP_ADDRESS>         按指定IP过滤，可多次指定
    -port <PORT_NUMBER>      按指定端口过滤，可多次指定
    -y <FILETYPE>            输出文件类型:
        L7                   仅保存应用层payload
        pcap                 保存完整PCAP帧 (默认)
    -recursive               递归处理输入目录下的所有抓包文件
```

### 7.2 新增扩展参数
```
    --mode <MODE>            运行模式: split(拆分模式, 默认) / archive(归档模式) / extract(提取模式)
    --archive <FILE>         归档模式下指定输出.rsplit文件路径
    --list-flows             提取模式下列出归档文件中所有流的元数据
    --extract <FLOW_ID>      提取指定ID的流，输出到标准输出或-o指定的文件
    --filter-ip <IP>         提取模式下按IP筛选流，可多次指定
    --filter-port <PORT>     提取模式下按端口筛选流，可多次指定
    --filter-proto <PROTO>   提取模式下按协议筛选流 (tcp/udp/icmp)
    --no-secondary-index     归档时不生成二级索引，减小文件体积
    --threads <NUM>          指定工作线程数，默认使用CPU核心数
    --no-mmap                禁用内存映射
    --verbose                输出详细日志
```

### 7.3 典型使用示例
#### 兼容原有场景
```bash
# 按会话拆分pcap文件
rsplitcap -r dumpfile.pcap -s session

# 按IP对拆分，输出到指定目录
rsplitcap -r dumpfile.pcap -s hostpair -o session_directory

# 过滤指定IP和端口，输出单个文件
rsplitcap -r dump.pcap -ip 1.2.3.4 -port 80 -port 443 -s nosplit

# 按1小时分片
rsplitcap -r dump.pcap -s seconds 3600

# 管道输入处理
tcpdump -n -s0 -U -i eth0 -w - | rsplitcap -r - -s session
```

#### 新增归档与提取场景
```bash
# 将pcapng按会话生成rsplit归档文件（含二级索引）
rsplitcap -r large.pcapng --mode archive --archive output.rsplit -s session

# 生成归档时不包含二级索引，追求极致写入速度
rsplitcap -r large.pcap --mode archive --archive output.rsplit --no-secondary-index

# 列出归档中的所有流
rsplitcap --mode extract --archive output.rsplit --list-flows

# 提取ID为123的流为独立pcap文件
rsplitcap --mode extract --archive output.rsplit --extract 123 -o flow_123.pcap

# 批量提取所有80端口的流到指定目录
rsplitcap --mode extract --archive output.rsplit --filter-port 80 -o extracted_flows/

# 提取指定IP+端口的所有流
rsplitcap --mode extract --archive output.rsplit --filter-ip 192.168.1.100 --filter-port 443 -o result/
```

## 11. 风险与应对
1. **PCAP-NG 格式复杂度高**
   - 风险：PCAP-NG 支持多种块类型与扩展特性，全量实现成本高
   - 应对：优先实现最常用的增强包块（EPB）、接口描述块（IDB）、节头块（SHB），覆盖 99% 实用场景；高级特性后续迭代补充

2. **超大文件流数量爆炸**
   - 风险：骨干网级抓包流数量达百万级，偏移列表内存占用过高
   - 应对：偏移采用增量编码降低内存占用；支持 LRU 淘汰将不活跃流的偏移列表暂存到临时文件；提供参数供用户根据场景调优

3. **随机读提取性能**
   - 风险：超大流提取时大量随机读可能降低性能
   - 应对：依赖操作系统页缓存与 mmap 预读优化；可选增加「流物化」功能，将高频访问的流导出为连续 Blob；绝大多数场景下随机读开销远小于磁盘IO，可接受

4. **索引体积膨胀**
   - 风险：小包场景下数据包数量极多，索引占比上升
   - 应对：Delta + LEB128 压缩将索引体积控制在原始流量的 0.5% 以内；提供关闭二级索引的选项，进一步减小体积
